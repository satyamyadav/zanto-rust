# Spec: Finance v0.2 — "Trustworthy Ledger"

**Date:** 2026-06-18  ·  **Status:** Draft (awaiting approval — no code yet)
**Source plan:** [docs/product/finance-next-version-plan.md](../product/finance-next-version-plan.md) Phases 0 + 1
**Goal:** Make the finance numbers correct (income vs expense) and the ledger editable + safe to import into.

---

## 1. Scope

**In:**
- Phase 0: a mutable data layer (`update`/`delete`/`get` by id) + transaction schema v2 with read-time back-compat.
- Real money model: `income` vs `expense`; cash-flow and balance computed correctly.
- Editable ledger: edit / recategorize / delete a transaction from the UI and via tools.
- Import dedupe: re-running a statement import adds zero duplicates.
- Category enforcement: validate against the profile's categories + a minimal merchant→category rules map; unknowns land in an "uncategorized" review filter.

**Out (later phases, do not build):**
- `transfer` type and multiple accounts (Phase 3 — transfers are meaningless without accounts).
- Budgets, recurring detection, trends (Phase 2). Statement-file parser pipeline (Phase 3). Goals/forecast (Phase 4).

## 2. Data layer (Phase 0)

`crates/zanto-core/src/data/mod.rs` — add three methods to `DataStore` (table is already `id INTEGER PRIMARY KEY AUTOINCREMENT, data TEXT, created_at`):

```rust
/// Replace a record's JSON by id. Errors if no row matched.
pub fn update(&self, store: &str, id: i64, record: &Value) -> Result<(), DataError>;
/// Delete a record by id. Ok even if it was already gone (idempotent).
pub fn delete(&self, store: &str, id: i64) -> Result<(), DataError>;
/// Fetch one record by id.
pub fn get(&self, store: &str, id: i64) -> Result<Option<Record>, DataError>;
```

- `update`: `UPDATE {table} SET data = ?1 WHERE id = ?2`; if `changes() == 0` return a new `DataError::NotFound { store, id }`.
- `delete`: `DELETE FROM {table} WHERE id = ?1`; never errors on missing.
- `get`: reuse the `query` row-mapping; `SELECT ... WHERE id = ?1`.
- Add unit tests: insert→update→get reflects change; delete→get is None; update missing id → `NotFound`.

These are generic (reused later for profile/widgets cleanup; out of scope here).

## 3. Transaction schema v2

`crates/zanto-desktop/src-tauri/src/apps/finance/mod.rs`.

A transaction record becomes:

```jsonc
{
  "type": "expense",          // "income" | "expense"   (default "expense")
  "amount": 12.50,            // always POSITIVE; sign derives from `type`
  "merchant": "Cafe",
  "category": "dining",       // validated; "uncategorized" if unknown
  "date": "2026-06-18",       // YYYY-MM-DD; defaults today
  "note": "",                 // optional
  "source": "manual",         // "manual" | "import"
  "import_hash": "..."        // present only when source == "import"
}
```

**Back-compat (no destructive migration):** existing rows have only `{date, amount, merchant, category}`. Normalize at read time with a single helper used by every aggregation:

```rust
/// Normalized view of a stored transaction, defaulting legacy/missing fields.
struct Txn { kind: TxnKind, amount: f64, category: String, date: String }
fn normalize_txn(v: &Value) -> Txn {
    // kind: missing → Expense; amount: coerce_amount (already exists) → abs();
    // category: string or "uncategorized"; date: string or "".
}
enum TxnKind { Income, Expense }
```

`amount` is stored positive; income/expense is carried by `type`. `coerce_amount` (added this session) stays the input coercion; apply `.abs()` so a stray "-12" doesn't flip a sign the `type` already owns.

## 4. Aggregation changes

Rewrite `compute_overview` and `compute_monthly_summary` to use `normalize_txn`:

`compute_overview` returns (additive — keep existing keys for the dashboard, add new):
```jsonc
{
  "empty": false,
  "balance": <lifetime income − lifetime expense>,
  "month": "2026-06",
  "income": <this-month income>,
  "month_total": <this-month expense>,          // unchanged meaning: spend
  "net_cash_flow": <income − expense, this month>,
  "transaction_count": <n>,
  "top_categories": [ ... expenses only ... ],
  "uncategorized_count": <expenses with category "uncategorized">,
  "series": { "labels": [...6 months], "data": [...monthly EXPENSE...] }
}
```

`compute_monthly_summary` returns:
```jsonc
{ "month": "YYYY-MM", "income": N, "total": <expense>, "net": <income−expense>, "by_category": [ ... expenses only ... ] }
```
`monthly_summary.svelte` shows income, spend, net, and the per-category expense breakdown.

`compute_transactions` MUST include each row's `id` so the UI can edit/delete:
```rust
.map(|r| { let mut o = r.data; o["id"] = json!(r.id); o })
```
(Today it returns `r.data` and drops the id — blocking edit/delete.)

## 5. Tools & actions (agent + UI share the deterministic flows)

`agent_tools()` / `dispatch_tool()` / `action()` in finance `mod.rs`:

1. **`add_transaction` v2** — add `type` (enum income|expense, default expense), `note`, keep lenient amount + abs. `required: ["amount"]`. On insert, apply category resolution (§6) and, when `source=="import"`, compute `import_hash` and **skip if a row with that hash exists** (return `{ "status": "duplicate_skipped" }`).
   - `import_hash = sha-less stable hash of` `format!("{date}|{amount:.2}|{merchant_lowercased}")` (use a simple `DefaultHasher` → hex; collisions tolerable for dedupe).

2. **`update_transaction`** (new) — args `{ id, ...any of type/amount/merchant/category/date/note }`. Loads via `data.get`, merges provided fields (re-resolving category if merchant/category changed, re-coercing amount), `data.update`. Returns the updated record. Registered as both an agent tool and a UI `action`.

3. **`delete_transaction`** (new) — args `{ id }`. `data.delete`. Returns `{ "status": "deleted", "id": id }`. Agent tool + UI action.

Tool descriptions stay one-line and direct (lesson from the chart tool: narrow, lenient, one-shot).

## 6. Category enforcement + rules

- The active profile already carries `categories: [..]`. On add/update, resolve category:
  1. If caller passed a `category` that is in the profile list → use it.
  2. Else if a **rule** matches the merchant → use the rule's category.
  3. Else → `"uncategorized"`.
- **Rules store** `finance_category_rules`: records `{ "merchant_contains": "starbucks", "category": "dining" }`, matched case-insensitively on merchant substring. Seeded empty; add a `save_category_rule` action + a tiny UI in the dashboard's edit mode (list + add/remove). (Keep minimal — full rules UX is a stretch goal.)
- **Review queue:** the transactions table gains an "Uncategorized" filter chip (drives a `query_transactions` with `category = "uncategorized"`), so the user can fix them in place.

## 8. Acceptance criteria

1. Logging income £2000 and an expense £12.50 → **Balance = 1987.50**, Income(mo)=2000, Spent(mo)=12.50, Net(mo)=1987.50. (Today balance would wrongly read 2012.50.)
2. A transaction can be recategorized and deleted from the table; the dashboard updates.
3. Re-running the same import (same date+amount+merchant) adds **zero** new rows (`duplicate_skipped`).
4. An add with a category not in the profile and no matching rule → stored `uncategorized`; appears under the Uncategorized filter; a rule `merchant_contains:"cafe"→dining` makes the next matching add land in `dining`.
5. Legacy transactions (pre-v2, no `type`) are treated as expenses and still aggregate correctly.
6. `cargo build` + `cargo test` (new data-layer + finance tests) + `pnpm check` + `pnpm build:web` all green.

## 10. Resolved decisions (locked 2026-06-18)
- **Q1 → One tool with `type`.** `add_transaction(type: "income"|"expense", …)`; no separate `add_income`. Fewer tools = more reliable for weak models.
- **Q2 → Full rules UI in v0.2.** Ship the merchant→category rules editor (list + add/remove) in the dashboard edit mode, in addition to the resolver + `finance_category_rules` store + the Uncategorized review filter. (T4 expands to include the rules editor component.)
- **Q3 → Display-only currency.** Format amounts using `profile.currency` (e.g. `GBP → £12.50`); single currency, no FX. Add a small `formatCurrency(amount, currency)` util shared by the table / KPIs / monthly_summary. No per-transaction currency.
