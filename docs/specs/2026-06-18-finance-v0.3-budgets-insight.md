# Spec: Finance v0.3 — "Budgets & Insight"

**Date:** 2026-06-18  ·  **Status:** Draft (awaiting decisions — no code yet)
**Source plan:** [docs/product/finance-next-version-plan.md](../product/finance-next-version-plan.md) Phase 2
**Builds on:** v0.2 "Trustworthy Ledger" (income/expense model, editable ledger, categories).
**Goal:** Turn the ledger into a coach — budgets with over/under signal, recurring-charge detection, and month-over-month trends — so there's a reason to open it weekly.

---

## 1. Scope

**In:**
- Per-category monthly **budgets** + budget-vs-actual (spent / limit / % / over) for the current month.
- **Overspend nudge** when a category exceeds its budget.
- **Recurring / subscription detection** (same merchant + ~amount, monthly cadence) → a Subscriptions surface.
- **Month-over-month** deltas + a per-category **trend** line chart.
- Surface all of the above (widgets and/or tabs) and reflect them in the "Monthly review".

**Out (later phases):** multiple accounts / net worth (Phase 3), statement-file import pipeline (Phase 3), goals + forecast (Phase 4). All aggregation stays deterministic Rust.

---

## 2. Budgets (backend)

New store `finance_budgets` (single latest-wins record, mirroring widgets/profile):
```jsonc
{ "budgets": [ { "category": "dining", "limit": 200 } ], "saved_at": 1718... }
```
- `get_budgets(data) -> { budgets: [...] }` (latest by `saved_at`; default `[]`).
- action `save_budgets` args `{ budgets: [{category, limit}] }` — keep only entries with a known/non-empty category and a positive numeric `limit` (coerce via `coerce_amount`).
- query `budgets`.

Budget-vs-actual is computed in `compute_overview` (this month, expenses only) and added to the result:
```jsonc
"budget_status": [ { "category": "dining", "limit": 200, "spent": 240, "pct": 1.2, "over": true } ],
"over_budget": [ { "category": "dining", "limit": 200, "spent": 240, "by": 40 } ]
```
`budget_status` lists every budgeted category (spent defaults 0); `over_budget` is the subset with `spent > limit` (drives the nudge).

---

## 3. Recurring detection (backend)

Pure helper `detect_recurring(records: &[Record], now_month: &str) -> Vec<Value>`:
- Consider **expenses** only. Key by `(merchant_lowercased, amount rounded to nearest whole unit)`.
- A group is "recurring" when it has **≥ 3 occurrences** across **≥ 3 distinct months**, with a median gap of ~25–35 days (monthly cadence). (Thresholds named as consts so they're tunable.)
- Return `[{ merchant, amount, count, last_date, monthly_total }]`, sorted by `monthly_total` desc.
- query `recurring -> { items: [...] }`. Unit-tested on synthetic records (a monthly Netflix-like charge is detected; a one-off is not).

---

## 4. Trends (backend)

Pure helper over the last 6 months (reuse `last_n_months`):
- Per-category monthly **expense** series: `{ months: [..6], categories: [ { category, data: [..6] } ] }` (top ~5 categories by total).
- Overall **month-over-month**: `mom_delta = thisMonthExpense − lastMonthExpense`, `mom_pct`.
- query `trends -> { months, categories, mom_delta, mom_pct }`. Also add `mom_delta`/`mom_pct` to `compute_overview` for a dashboard indicator. Unit-tested.

---

## 5. Frontend

`crates/zanto-desktop/src/lib/apps/finance/`:

- **Budgets editor** `Budgets.svelte` (shown in dashboard edit mode, beside CategoryRules): list profile categories with a numeric limit input each; Save → `save_budgets`. Loads `budgets` query.
- **Budget-vs-actual** on the dashboard: a section (or a new widget `kind:"budget"`) rendering each `budget_status` row as a labeled progress bar — `text-success` under, `text-warning` near (≥80%), `text-destructive` over. Currency-formatted via the existing `formatCurrency`.
- **Overspend nudge:** when `overview.over_budget.length > 0`, a banner like "Over budget in dining by £40" (sums multiple), using `bg-destructive/10 text-destructive`. (See Decision Q2 for native notification.)
- **Subscriptions** surface (Decision Q3): list from `recurring` — merchant, amount, "monthly", last seen; a total "£X/mo in subscriptions".
- **Trends:** a per-category line chart (reuse `Chart.svelte`, `type:"line"`, one dataset per category from `trends`) + a MoM indicator near the KPIs ("Spending ↑ £120 (15%) vs last month", up = `text-destructive`, down = `text-success`).
- **WidgetBuilder + default_widgets:** add sources `budget` (budget-vs-actual) and `trends` (category trend line). Add a MoM KPI source if surfaced as a KPI.
- **Monthly review:** extend `monthly_summary.svelte` (and the "Monthly review" start-action prompt) to include over-budget categories and the MoM delta.

## 6. Acceptance criteria
1. Set a £200 dining budget; spend £240 → budget bar shows over (red), dashboard shows "Over budget in dining by £40".
2. Three monthly £9.99 "Netflix" expenses → appears under Subscriptions as ~£9.99/mo; a single coffee does not.
3. Trends line chart shows per-category last-6-months; MoM indicator matches (this − last month expense).
4. Budgets persist across reloads; editing a budget updates the bars.
5. `cargo build` + `cargo test` (new budget/recurring/trends unit tests) + `pnpm check` + `pnpm build:web` green.

Dependencies: T1→T4; T2→T5; T3→T5; then T6; then T7. (T1–T3 backend can land in any order.)

## 8. Resolved decisions (locked 2026-06-18)
- **Q1 → Per-category limits.** Monthly limit per category; overall = the sum (derived, no separate cap).
- **Q2 → In-app banner + native notification.** Avoid App-trait surgery: add a generic `notify(title, body)` IPC command (reuses the existing `NotificationExt` path used for "Reply ready"). The dashboard, on overview load, fires it once per over-budget category per month (deduped in `localStorage` keyed by `YYYY-MM|category`), alongside the always-on in-app banner.
- **Q3 → Both.** Budget / Subscriptions / Trends as composable dashboard **widgets** AND dedicated **Subscriptions** and **Trends** tabs (alongside Dashboard / Transactions / Resources).
