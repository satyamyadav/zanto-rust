# Spec: Finance v0.4 — "Import & Accounts"

**Date:** 2026-06-18  ·  **Status:** Draft (awaiting decisions — no code yet)
**Source plan:** [docs/product/finance-next-version-plan.md](../product/finance-next-version-plan.md) Phase 3
**Builds on:** v0.2 (money model, dedupe) + v0.3 (budgets, insight).
**Goal:** Get many transactions in fast and reliably from a **file the user already has** (CSV/XLSX — no external service, ever), and organize money across the user's **own accounts** (checking / savings / card / cash) with per-account balances, transfers, and net worth.

**Constraint (explicit):** inputs are the **filesystem and chat only**. No bank APIs, no aggregation services, no network. "Import" = parse a local statement file. "Accounts" = internal labels for the user's own accounts.

---

## 1. Scope

**In:**
- **Statement import pipeline:** local CSV/XLSX → parsed rows → **review table** (column mapping + account + sign rule) → **batch insert** reusing the v0.2 `import_hash` dedupe.
- **Accounts:** a managed list (name + type + opening balance); each transaction tagged with an account; per-account balances; **net worth** (sum across accounts).
- **Transfers:** move money between the user's own accounts, neutral to income/expense.

**Out (Phase 4):** goals, debt payoff, cash-flow forecast. Still no external/live data. No multi-currency per account (v0.2's single display currency stands).

## 2. Tabular file parsing (zanto-core)

New `crates/zanto-core/src/tools/docs/parse_table.rs` exposing:
```rust
pub struct TableData { pub headers: Vec<String>, pub rows: Vec<Vec<String>> }
pub fn parse_table(path: &Path) -> Result<TableData, String>;
```
- CSV/TSV → the `csv` crate (add `csv = "1"` to zanto-core), first row = headers, fields as strings.
- XLSX/XLS/ODS → `calamine` (already a dep): first worksheet, first non-empty row = headers, cells stringified.
- Unsupported extension → `Err`. Cap returned rows at a sane max (e.g. 5000) and report if truncated.
- Unit tests on a small in-repo CSV fixture (headers + rows parse).

`parse_table` is a plain function (not an LLM tool) reused by the finance import flow. Permission to read the path is enforced by the caller (the finance IPC path runs `permissions.check` like the existing image/doc paths).

## 3. Import pipeline (finance backend)

New stores/fields and flows in `crates/zanto-desktop/src-tauri/src/apps/finance/mod.rs`:

- action/tool **`parse_statement`** args `{ path }` → `{ columns: [..headers], preview: [[..]] (first ~50 rows), row_count, suggested_mapping: { date, amount, merchant, category? } }`.
  - `suggested_mapping` heuristics (case-insensitive header match): date ← header contains "date"; merchant ← "description"|"merchant"|"payee"|"name"|"details"; amount ← "amount"|"value"|"debit"|"credit"|"total" or the first column that parses as numeric in the preview; category ← "category" if present.
- action **`import_transactions`** args `{ path, mapping: { date, amount, merchant, category? }, account, sign_rule }`:
  - Re-`parse_table(path)`, map each row via `mapping` (column header → field), build a transaction, and call the existing `do_add_transaction` with `source: "import"` and `account` (so `import_hash` dedupe applies). 
  - **sign_rule** (see Decision Q1) decides `type` from the amount: `"signed"` → negative ⇒ expense, positive ⇒ income; `"expense"` → every row an expense (sign ignored, amount abs).
  - Return `{ inserted, skipped (duplicates), errors: [{ row, reason }] }`.
- The IPC `run_app_action` path for these passes the file path through `permissions.check(path, Read)` before parsing (mirror `ipc/chat.rs` image handling). Add this guard in the finance IPC dispatch, not in the pure app (which has no permissions handle) — i.e. resolve+read in IPC, pass parsed rows down, OR expose a `permissions`-aware wrapper. **Decision Q3** covers where the guard lives.

## 4. Accounts (finance backend)

- New store `finance_accounts`, latest-wins record `{ accounts: [ { name, type, opening_balance } ], saved_at }` (type ∈ checking|savings|card|cash). `get_accounts`, `do_save_accounts` (action), query `accounts`. Seed a default `[{ name: "Cash", type: "cash", opening_balance: 0 }]` when none saved.
- Transaction gains an `account` field (string; defaults to the first saved account's name, else "Cash"). `add_transaction`/`update_transaction`/import all accept `account`.
- **Transfers** (Decision Q2 → single record): `type: "transfer"` with `account` (from) + `to_account` (to) + positive `amount`. A new `add_transfer` tool/action args `{ amount, from_account, to_account, date?, note? }`. `normalize_txn` gains `TxnKind::Transfer` (excluded from income/expense/net/month_total/series/budget/trends — it only moves money between accounts).
- **Per-account balance & net worth** (deterministic Rust):
  - `account_balance(name) = opening_balance + Σ(income to account) − Σ(expense from account) + Σ(transfer_in) − Σ(transfer_out)`.
  - `net_worth = Σ account_balance`.
  - Add to `compute_overview`: `accounts: [{ name, type, balance }]`, `net_worth`. Keep existing `balance` (lifetime income−expense, transfers excluded) for back-compat; the dashboard headline becomes **Net worth**.
- Pure helpers `compute_account_balances(accounts, records)` + extend `normalize_txn` — unit-tested (income+expense+transfer across two accounts yields correct per-account balances and a net worth that ignores transfers).

## 5. Frontend

`crates/zanto-desktop/src/lib/apps/finance/`:

- **Import tab** `Import.svelte` (new top-level tab): pick a file (reuse the existing file/attachment picker or a path input) → calls `parse_statement` → **review table**: shows columns with **mapping dropdowns** (Date / Amount / Merchant / Category → which column), an **account** selector, a **sign rule** toggle ("amounts are signed (− = spend)" vs "all are expenses"), and a preview of the first rows. "Import N rows" → `import_transactions` → toast "Imported X, skipped Y duplicates". 
- **Accounts** `Accounts.svelte` (tab + edit-mode editor): manage accounts (name, type, opening balance; add/remove) via `save_accounts`; show **per-account balance cards** and a **Net worth** headline.
- **Transfer** entry: a small form (in the Transactions or Accounts view) — amount, from, to — calling `add_transfer`.
- **Account column/selector:** TransactionsView gains an Account column + the inline editor gains an account `<select>`; the add flow defaults to the active/first account.
- **KPIs/widgets:** add `net_worth` KPI source; an `accounts` widget (per-account balances). Net worth replaces Balance as the default headline KPI; keep Balance available.
- Currency formatting via the existing `formatCurrency`.

## 6. Acceptance criteria
1. A 50-row CSV imports in one reviewed step; correct columns auto-suggested; re-importing the same file inserts **0** duplicates.
2. "All are expenses" sign rule stores every row as an expense; "signed" stores negatives as expenses and positives as income.
3. Two accounts (Checking opening £1000, Card opening £0): a £50 expense on Card and a £200 transfer Checking→Card → Checking £800, Card −£50 + £200 = £150… (balances reconcile); **net worth £950**; the transfer shows as neither income nor expense.
4. Accounts persist; editing opening balance updates net worth.
5. `cargo build` + `cargo test` (parse_table, import mapping, account balances) + `pnpm check` + `pnpm build:web` green.

## 8. Resolved decisions (locked 2026-06-18)
- **Q1 → Debit/credit columns (primary).** Import mapping is `{ date, merchant, category?, debit?, credit?, amount? }`. Per row: if `amount` is mapped, use the signed rule (negative ⇒ expense, positive ⇒ income); otherwise a value in the **debit** column ⇒ expense and a value in the **credit** column ⇒ income. This honors debit/credit statements while still handling single-column exports via the optional `amount` mapping. (Supersedes the §3 `sign_rule` wording.)
- **Q2 → Single transfer record.** `type:"transfer"` with `account` (from) + `to_account` (to) + positive `amount`; neutral to income/expense; one row to add/edit/delete.
- **Q3 → Both file sources, IPC-guarded.** The Import tab accepts a chat-attached file OR a filesystem path; the finance IPC layer resolves the path via `permissions.check(path, Read)` before `parse_table` reads it (the pure app never touches the filesystem directly).
